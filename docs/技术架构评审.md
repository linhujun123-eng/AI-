## 二、逐层评估

### 2.1 前端架构

| 维度 | 现状 | 评分 | 评价 |
|------|------|------|------|
| **框架选型** | React 19 + TS + Vite 5 | ✅ 优 | 主流方案，生态好，TS 零错误，build 稳定 |
| **状态管理** | Zustand 5 stores（player/ab-loop/stem/song/auth） | ✅ 优 | 轻量准确，每个 store 职责清晰，subscribe 模式适合音频同步 |
| **路由** | React Router 7, 2 routes | ✅ 优 | 简单够用，后续加页面零成本 |
| **样式** | CSS Modules + CSS Variables | ✅ 优 | 与 Neon 设计系统配合自然，隔离性好 |
| **音频引擎** | WaveSurfer.js (时间源) + Web Audio API (分轨) | ✅ 优 | 双引擎分工合理，WaveSurfer 管 UI+时间，Web Audio 管分轨混音 |
| **API 层** | 单文件 api.ts，全量封装 | 🟡 可改进 | 功能完整但文件越来越大（240 行），需要按领域拆分 |
| **代码量** | 29 个 TS/TSX 文件，338KB JS 产出 | ✅ 优 | 精炼，无冗余 |
| **测试** | 无 | 🔴 缺失 | 无单元测试、无 E2E，冷启动前需补关键路径 |
| **错误边界** | 无 | 🔴 缺失 | 无 React Error Boundary，组件崩溃整页白屏 |

**具体问题**：

1. **api.ts 膨胀** — 认证、歌曲、和弦、分轨、health 全在一个文件，Phase C 加埋点/练习记忆后会更大
2. **无全局错误处理** — fetch 失败、AudioContext 创建失败、WaveSurfer 加载失败 → 用户看到的是白屏或卡死
3. **预置歌曲硬编码** — `data/songs.ts` 硬编码 10 首曲目，扩到 20 首靠改源码，应改为后端/配置驱动
4. **无 loading/skeleton 状态** — 首页和练习页没有骨架屏，网络慢时体验差

### 2.2 后端架构

| 维度 | 现状 | 评分 | 评价 |
|------|------|------|------|
| **框架** | FastAPI 单文件 (~500 行) | 🟡 可改进 | MVP 够用，但 500 行单文件不可持续 |
| **数据库** | SQLite (WAL) 2 表 | 🟡 可改进 | 内测 ≤300 人够用，但有并发写锁风险 |
| **文件存储** | 本地文件系统 (audio/chords/stems) | 🟡 可改进 | 单机可用，无法水平扩展，无 CDN |
| **认证** | JWT + bcrypt | ✅ 优 | 标准实现，30 天有效期合理 |
| **异步处理** | FastAPI BackgroundTasks | 🟡 可改进 | 够用但不可靠：服务重启任务丢失，无重试 |
| **部署** | 无（dev 环境 uvicorn 直跑） | 🔴 缺失 | 无 Gunicorn/Supervisor/Docker/systemd |
| **日志** | print 语句 | 🔴 缺失 | 无结构化日志，无日志文件/轮转 |
| **配置** | 硬编码 + 少量 env | 🟡 可改进 | JWT_SECRET 有 env fallback，其余全硬编码 |

**具体问题**：

1. **单文件 500 行** — 路由、模型、认证、和弦识别、分轨全塞一起，难维护
2. **BackgroundTasks 不可靠** — 进程重启 = 正在分轨的任务丢失。当前有 `processing` 状态但无恢复机制，歌曲会永远卡在 `processing`
3. **并发安全** — SQLite 单写锁，两个用户同时上传可能冲突；`get_db()` 每次新建连接，无连接池
4. **无速率限制** — 注册、上传无 rate limit，被恶意刷可打爆
5. **CORS 硬编码** — 只允许 localhost:5173/5174，生产环境需要配置
6. **文件大小无分片** — 50MB 单次上传，移动端弱网容易超时
7. **分轨 subprocess 无控制** — 多个用户同时上传 → 并行跑多个 Demucs → CPU/内存爆炸

## 四、核心问题诊断（按严重度排序）

### 🔴 P0 — 冷启动阻塞项

#### 1. 后端单文件 + 无部署方案

**现状**：`chord_api.py` 是唯一后端文件（500+ 行），用 `python chord_api.py` 直接跑 uvicorn，无生产部署配置。

**风险**：
- 服务崩溃无自动重启
- 无日志持久化，出问题无法排查
- CPU 密集任务（CREMA/Demucs）阻塞 API 响应
- 单 worker = 同时只能处理一个请求的 CPU 密集任务

**建议**：
```
Phase A 最低可行方案：
  chord_api.py → 拆分为 routes/ + services/ + models/
  部署：systemd + gunicorn (多 worker) + nginx 反代
  CPU 任务：独立进程/队列（Celery 太重 → 轻量方案见下文）

Phase C 演进方案：
  Docker Compose: web(gunicorn) + worker(任务队列) + nginx
  或 直接上 fly.io / Railway
```

#### 2. 异步任务不可靠

**现状**：和弦识别（~90s）和分轨（~5min）用 FastAPI `BackgroundTasks` 执行。

**风险**：
- 服务重启 → 进行中的任务丢失 → 歌曲永远卡在 `processing`
- 多用户并发上传 → 同时跑 N 个 CREMA + N 个 Demucs → CPU/内存爆炸
- 无重试机制

**建议**：
```
Phase A（轻量方案，不引入 Celery/Redis）：
  1. 启动时扫描 status='processing' 的歌曲 → 重置为 'idle' 或自动重试
  2. 用信号量控制并发：同时最多 1 个 CREMA + 1 个 Demucs
  3. 超时检测：processing 超过 15min 自动标记为 error

Phase C（正式方案）：
  引入简单任务队列（如 huey/rq + Redis）或 SQLite 实现的穷人队列
```

#### 3. 预置歌曲硬编码

**现状**：`data/songs.ts` 硬编码 10 首歌的元数据，`public/audio/` 放静态文件。扩到 20 首要改代码 + 重新 build。

**建议**：
```
方案 A（简单）：保持前端硬编码但改为 JSON 配置文件动态 import，
  加曲目 = 加 JSON + 加音频文件 + rebuild
方案 B（推荐）：预置歌曲也走后端 API，有管理员角色可上传/管理预置曲目
  - songs 表加 is_preset / is_public 字段
  - 首页展示 = 预置(全部可见) + 用户上传(仅自己)
  - 预置歌曲的音频仍走 CDN/静态文件（性能更好）
```

### 🟡 P1 — 体验 & 可维护性

#### 4. 前端缺乏错误处理

**现状**：无 React ErrorBoundary，无全局 API 错误处理，AudioContext/WaveSurfer 失败无降级。

**建议**：
```
- App.tsx 外层包 ErrorBoundary → 显示友好错误页（含"刷新"按钮）
- api.ts 统一错误处理：401→自动跳登录，5xx→Toast 提示
- WaveSurfer 加载失败 → 降级为纯进度条模式
- AudioContext 创建失败 → 提示"请点击页面以启用音频"
```

#### 5. api.ts 需要拆分

**现状**：240 行，认证+歌曲+和弦+分轨+health 全在一起。

**建议**：
```
services/
  api/
    client.ts      — apiFetch, token 管理, 错误处理
    auth.ts        — register/login/getMe
    songs.ts       — CRUD + audio URL
    chords.ts      — fetchChords/triggerRecognition
    stems.ts       — triggerSeparation/getStemAudioUrl
    events.ts      — 埋点上报（Phase A 新增）
    practice.ts    — 练习记忆（Phase A 新增）
```

#### 6. 轮询 → WebSocket/SSE

**现状**：和弦状态 2s 轮询，分轨状态 5s 轮询。300 个用户每人上传一首歌 = 600 个轮询循环。

**建议**：
```
Phase A：保持轮询（用户少，够用），但加超时退出机制
Phase C：SSE（Server-Sent Events）推送状态变更
  - 比 WebSocket 简单，单向推送够用
  - FastAPI 原生支持 StreamingResponse
```

### 🟢 P2 — Phase C 再处理

#### 7. SQLite 并发限制

**现状**：每次请求 `get_db()` 新建连接，无连接池。SQLite 单写锁。

**评估**：300 人内测阶段，同时在线 ≤30 人，写操作低频（上传/注册偶发）。**SQLite 够用，但需要**：
- 连接池（WAL 模式已开，多读者 OK）
- 写操作加锁或序列化
- Phase C 超过 500 DAU 再考虑 PostgreSQL

#### 8. 音频文件无 CDN

**现状**：所有音频通过 FastAPI `FileResponse` 流式返回。

**评估**：内测阶段服务器带宽可能撑 50-100 并发流式播放（取决于服务器带宽），但到 Phase C 需要：
- 预置曲目 → CDN/OSS 静态托管
- 用户上传 → 服务器代理 或 签名 URL 直传 OSS

## 七、结论

### 总体评价：**架构合理，MVP 阶段的好选择，但需要在冷启动前做结构性加固**

**做得好的**：
- 前端技术栈选型精准（React + Zustand + WaveSurfer + Web Audio），没有过度工程
- 双音频引擎（WaveSurfer 时间源 + Web Audio 分轨）设计巧妙
- API 设计清晰一致
- CSS Modules + Variables 与 Neon 设计系统配合自然
- 29 个文件，338KB 产出，代码精炼

**核心风险**：
- 后端单文件 + BackgroundTasks 在多用户场景下是隐患
- 无部署方案 = 无法上线
- 无错误处理 = 真实用户会遇到白屏

**好消息**：所有问题都是"加固"类型的——不需要推倒重来，只需要在现有架构上补齐缺失的层。预计 **3-4 天的架构加固**就能让系统达到冷启动 300 人的可靠度。
